# Positioning Guide

Use company research and JD signals to decide how to frame the resume. The same background can be framed five different ways. The facts do not change — the emphasis does.

---

## Company size and stage → resume tone

| Company type | What they value | Resume tone |
|---|---|---|
| Early startup (seed to Series A, <50 people) | Builders, ownership, shipping without being asked | Direct, specific, decisions over actions |
| Growth stage (Series B/C, 50-500 people) | Fast learners, structured thinkers, proven delivery | Builder identity + some process awareness |
| Large tech company (500+ people, MNC) | Communication, CS fundamentals, structured process | Professional, thorough, academically anchored |
| Traditional corporate / IT services (TCS, Infosys, Wipro) | CGPA, aptitude, willingness to learn, flexibility | Academic-first, clean, no startup jargon |
| Manufacturing/industrial company | Engineering degree, domain basics, reliability | Conservative, qualification-led |

---

## Section order by role type

**Large corporates and MNCs (TCS, Sabre, Infosys, manufacturing):**
Education → Skills → Experience → Projects

Reason: HR reads hundreds of resumes. They filter by CGPA first. Put education where they look first.

**Startups and product-adjacent roles:**
Summary → Projects → Experience → Skills → Education

Reason: Founders care about what you built. Projects prove it faster than job titles.

**GTM / Sales roles:**
Summary → Experience → Skills → Education

Reason: Sales hiring managers read experience first. Numbers, markets, and outcomes up front.

**GTM Engineer / Forward Deployed:**
Summary → Projects → Experience → Skills → Education

Reason: Equal weight on building and selling. Projects show the technical half; experience shows the sales half.

**Analytics / Data roles:**
Summary → Education → Experience → Projects → Skills

Reason: Academic scores and CS background are the first filter. Experience shows application.

**AI Engineering / Technical delivery:**
Summary → Experience → Projects → Skills → Education

Reason: Delivery track record dominates. Projects back it up. Education is secondary.

## What to emphasise per role type

### GTM / Sales roles
- Lead with market exposure (which countries, what volume)
- Cold outreach, closing, and pipeline numbers if available
- Avoid heavy technical framing — it dilutes the sales signal
- Tools: Apollo, LinkedIn Sales Navigator, CRM tools

### GTM Engineer / Forward Deployed
- Name AI coding tools explicitly (companies hiring for this often value them)
- Lead generation + automation together
- Specific tool names and API integrations
- Show both the sales and the build side

### Product / Engineering hybrid (PM-adjacent)
- Projects go first and get the most space
- Architecture decisions matter more than tool lists
- "Why" beats "what" — explain choices, not just actions
- Documentation and communication signals belong in bullets

### Analytics / Data roles
- Reframe pipeline work as data engineering
- Name data sources, output formats, accuracy metrics
- Customer delivery and feedback loops map to their "calibration" language
- Avoid GTM or sales framing entirely

### AI Engineering
- Technical stack breadth matters
- Deployment, testing, and client calibration are all strong signals
- Lead with scale: number of systems shipped, number of clients
- Voice agents, SMS agents, RAG pipelines — name the categories

### Campus / Large IT (TCS, Infosys, etc.)
- CGPA is the single most important signal — lead with it
- "No active backlogs" is worth stating explicitly
- Avoid startup and founder language — it reads as a flight risk
- Technical skills section should name CS fundamentals (even if basic)
- Projects are secondary but should be framed in technical delivery terms

## CGPA/percentage: when to lead vs de-emphasise

| Score | Traditional corporate / IT services / PSU / govt track | Startups / product / GTM |
|---|---|---|
| ≥7.0 CGPA / ≥70% | State plainly, lead with it if targeting mass-recruiter service companies (many hard-filter on this before round 1) | State plainly, no special emphasis |
| <7.0 CGPA / <70% | Still state it (many of these companies screen on it as an eligibility gate elsewhere in the pipeline regardless of the resume, so omitting looks evasive) but do not lead the section with it — put projects/skills ahead of education in layout order instead | De-emphasise or omit; these companies rarely screen on it, and it adds nothing |

Never estimate or round a score to look better. If CGPA-to-percentage conversion is asked for, only convert using the institution's official formula (see `INTERVIEW.md`).

## Declaration line

A closing line ("I hereby declare that the information provided above is true and correct to the best of my knowledge.") is a distinctly Indian resume convention with no US equivalent.

- **Include** for: traditional corporate / IT services, PSU, and government hiring tracks — still expected there.
- **Omit** for: startups, product companies, GTM, and any founder-led or modern-hiring context — reads as dated filler and wastes space on a one-page resume.

## Handling the skills section honestly

**If they built using an AI coding tool:**
Do not list the language as a core skill. Options:
- List the AI tool (Claude Code, Cursor, GitHub Copilot) as a tool
- Write "Familiar with Python / JavaScript" if they can read and understand the code
- Focus skills section on tools, platforms, and systems they use directly

**If they have real coding ability:**
List languages with context: "Python (data pipelines, scripting)" not just "Python"

**If they have no coding ability:**
Skills section focuses on: tools, platforms, AI systems, outreach tools, productivity tools, documentation

---

## What to leave out

- Any skill they cannot demonstrate in a 5-minute conversation
- Market experience they do not have (do not list UK if they only sold to US and Australia)
- Founder or co-founder framing if it will raise "why are you joining us?" questions — use "Independent Consultant" instead
- Certifications with no credential backing
- Generic soft skills ("team player," "hard worker," "detail-oriented") — these add nothing

## Page count decision

| Situation | Page count |
|---|---|
| Thin experience, 1-2 projects | 1 page (mandatory) |
| Dense projects + 1-2 strong roles | 1 page if it fits without cutting substance |
| Dense projects too long for 1 page | 2 pages, but only if page 2 is 50%+ full |
| Applying to large corporate / ATS-heavy process | 1 page strongly preferred |
| Applying to founder-led startup | 1-2 pages acceptable if content justifies it |

A sparse second page is always worse than a tight first page.
